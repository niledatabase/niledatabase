## Installation

`@niledatabase/client` exposes the the client side functions for various authorization requests. It handles API requests for sign-in, sign-up, sessions, and password reset. The client is designed to work with `@niledatabase/server` which provides the server-side routes.

```bash
npm install @niledatabase/client
```

## Getting Started

For most cases, These helpers call the underlying singleton instance of `Authorizer`, which can be accessed via `auth`

```ts
import {
  auth
  // signIn // could just use this instead
} from '@niledatabase/client';

await auth.signIn('credentials', { email: 'user@example.com', password: 'pw' });
```






### getSession
Fetches the current session. If a session is cached and still valid it returns that, otherwise it requests `/auth/session`.

```ts
await auth.getSession();
```

### signIn
Sends a sign-in request to the authentication endpoint. The initial signIn values are [the supported providers for single sign on](../../../singlesignon/google), in addition to `credentials`

```ts
await auth.signIn('credentials', {
  email: 'user@example.com',
  password: 'password',
});
```
#### Params
| Name      | Type        | Description                                   |
|-----------|-------------|-----------------------------------------------|
| `callbackUrl` | `string`    | The URL to return when the request is complete.             |
| `redirect`| `boolean`    | Allows configuration of client-side redirects in the case of credential login        |

### signOut
Logs the user out and clears the session.

```ts
await auth.signOut({ callbackUrl: '/goodbye' });
```

### signUp
Creates a new user account. You can also create tenants during sign up. A boolean value will create a tenant with the same email address. If a string is passed, that will be the name of the created tenant.

```ts
await auth.signUp({
  email: 'user@example.com',
  password: 'pass',
  createTenant: true,
});
```

### resetPassword
Sends requests to the password reset endpoints. You must be an authorized user to do this action.

```ts
await auth.resetPassword({
  email: 'user@example.com',
  password: 'new-pass',
});
```

### getCsrfToken
Returns a CSRF token used for form submissions.

```ts
const token = await auth.getCsrfToken();
```

### getProviders
Returns a list of available authentication providers.

```ts
const providers = await auth.getProviders();
```

## Authorizer
You can also customize your own authorizer on the client by creating an instance of `Authorizer`.


```ts
import { Authorizer } from '@niledatabase/client';

const auth = new Authorizer({ baseUrl: 'https://myapp.com' });
```
### Constructor Options

`new Authorizer(config)` accepts the following options:

| Name      | Type        | Description                                   |
|-----------|-------------|-----------------------------------------------|
| `baseUrl` | `string`    | The base URL of your application.             |
| `basePath`| `string`    | Optional API path, defaults to `/api`.        |
| `init`    | `RequestInit` | Default `fetch` options for all requests.    |

The instance exposes a `state` object with `baseUrl`, `session`, and loading status. You can update the configuration later by calling `auth.configure({ baseUrl })`.

## Multi-factor (Client)

Client helper for MFA setup, challenge verification, and removal against `/auth/mfa`.

### Overview
Use `mfa` from `@niledatabase/client` when you need to script MFA flows directly or pair with custom UI (React components wrap this helper under the hood). It mirrors the API contract: `POST` starts setup, `PUT` verifies a challenge, and `DELETE` disables an enrolled method when `remove: true` is supplied. Redirect responses are surfaced as `{ url: string }` so you can route accordingly.

<CodeGroup>
```ts setup
import { mfa } from '@niledatabase/client';

const setup = await mfa({ method: 'authenticator', scope: 'setup' });
```

```ts verify
const verified = await mfa({
  token: setup.token,
  code: '123456',
  method: setup.method,
  scope: setup.scope, // "setup" or "challenge"
});
```

```ts disable
await mfa({ method: 'authenticator', remove: true });
```

```ts challenge
// Finish an MFA-required sign-in
const challenge = await auth.signIn('credentials', { email, password });
if (challenge && typeof challenge === 'object' && 'token' in challenge) {
  const result = await mfa({
    token: String((challenge as any).token),
    code: '654321',
    method: (challenge as any).method ?? 'authenticator',
    scope: 'challenge',
  });
  if (!result?.ok) throw new Error('MFA verification failed');
}
```
</CodeGroup>

### API
`mfa(params: MfaParams)` automatically selects the HTTP verb based on the payload.

| Name | Type | Default | Description |
| --- | --- | --- | --- |
| `token` | `string` | `undefined` | Challenge token from setup or sign-in; required for verification and some email disable flows. |
| `scope` | `'setup' \| 'challenge'` | `'challenge'` | Whether you are initiating setup or completing a challenge. |
| `method` | `'authenticator' \| 'email'` | `'authenticator'` | MFA mechanism to target. |
| `code` | `string` | `undefined` | 6-digit or recovery code when verifying or disabling. |
| `remove` | `boolean` | `false` | Sends a `DELETE` request to disable MFA for the current user. |
| `baseUrl` | `string` | Browser origin | Override the base API URL when not using the default proxy. |
| `fetchUrl` | `string` | `${baseUrl}/api/auth/mfa` | Full URL override for the MFA endpoint. |
| `init` | `RequestInit` | `undefined` | Extra fetch options (credentials, headers, etc.). |
| `auth` | `Authorizer` | `undefined` | Reuse an existing `Authorizer` instance/context. |

Returns one of:
- A setup payload (`AuthenticatorSetup` or `EmailSetup`) with `token`, `scope`, and method-specific fields.
- `{ ok: true; scope: 'setup' \| 'challenge'; recoveryCodesRemaining?: number }` after verification.
- `{ url: string }` when the backend expects a redirect (inspect `error` query param for messaging).

### Error handling and notes
- Non-200 responses are coerced into `{ url: string }` with an `error` search parameter; parse it to display user-friendly errors.
- Codes should be 6 digits for authenticator/email verification; recovery codes are string tokens issued during setup.
- Challenge tokens expire; expect 410 for stale tokens and 404 for missing challenges.
- Email MFA may require the same token for verification and disable flows.
